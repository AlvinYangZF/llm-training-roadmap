Overview Transformer Architecture

The Transformer processes an input sequence through embedding, multiple attention + feed-forward layers, and a final output projection. Below is the data flow for a decoder-only model (GPT-style).

Input Tokens → Token Embedding + Positional Encoding
↓
× N Transformer Blocks
↓
Layer Normalization
↓
Multi-Head Self-Attention (Q, K, V)
↓
Residual Add + Layer Normalization
↓
Feed-Forward Network (Linear → GELU → Linear)
↓
Residual Add + Layer Normalization
↓
End of block — repeat × N
↓
Linear Projection → Softmax → Next Token Probabilities

Details Key Components

1. Embedding Layer

Each input token is mapped to a dense vector (e.g. 768-d). A positional encoding is added so the model knows token order — Transformers have no inherent sense of sequence position.

2. Multi-Head Self-Attention

The core mechanism. Input is linearly projected into Q (Query), K (Key), and V (Value) matrices. Attention scores determine how much each token attends to every other token. Multiple heads allow attending to different patterns simultaneously.

3. Feed-Forward Network

A two-layer MLP applied independently to each position: expands dimensionality (e.g. 768 → 3072), applies GELU activation, then projects back down (3072 → 768).

4. Residual Connections & Layer Norm

Every sub-layer has a skip connection (output = sublayer(x) + x) and layer normalization. This stabilizes training and enables very deep networks (dozens of layers).

Inference Two-Phase Generation

LLM text generation happens in two distinct phases with very different compute characteristics.

Phase 1: Prefill

Process the entire prompt in one forward pass.

  • All prompt tokens processed in parallel
  • Compute-bound — heavy matrix multiplications
  • Builds the KV Cache for all prompt tokens
  • Produces the first output token
  • Time ∝ prompt length

Phase 2: Decode (Autoregressive)

Generate tokens one at a time, each depending on all previous.

  • Only 1 new token per forward pass
  • Memory-bound — reads large KV cache, little compute
  • Appends new K, V to the KV Cache
  • Repeats until EOS or max length
  • Time ∝ output length

Deep Dive KV Cache

The KV Cache is the key optimization that makes autoregressive decoding efficient.

Without KV Cache

Each new token would require recomputing attention over all previous tokens from scratch — O(n²) per generated token, O(n³) total.

With KV Cache

Store the K and V projections from all previous tokens. When generating a new token, only compute Q for the new token and attend to the cached K/V. Cost per token: O(n) attention, O(n²) total for the full sequence.

Cache size = 2 × n_layers × seq_len × d_model × dtype_bytes
Example: 32 layers × 4096 seq × 4096 dim × 2 (K+V) × 2 bytes (FP16) ≈ 2 GB per request

Core Insight FlashAttention — IO-Aware Attention

Standard attention is bottlenecked by memory bandwidth, not compute. FlashAttention restructures the computation to minimize data movement between slow HBM and fast SRAM.

SRAM (On-chip Cache)
Size: ~20 MB  |  Bandwidth: 19 TB/s  |  Super fast
↕ Data transfer is the bottleneck (13× slower)
HBM (GPU Main Memory)
Size: 40–80 GB  |  Bandwidth: 1.5 TB/s  |  Large but slow

Problem Standard vs FlashAttention

Standard Attention

Every intermediate matrix is written to and read from HBM:

Q,K in HBM → S=QKT → write to HBM
S in HBM → P=softmax(S) → write to HBM
P,V in HBM → O=PV → write to HBM

Memory IO: O(N²) reads + O(N²) writes × 3 passes
Materializes full N×N attention matrix in HBM

FlashAttention

Process in small tiles entirely within SRAM — never materialize N×N matrix:

for each block of Q:
  for each block of K,V:
    compute S,P in SRAM
    accumulate O (online softmax)

Memory IO: O(N²d / M) where M = SRAM size
Only reads Q,K,V once, writes O once to HBM

Key Trick Online Softmax (Tiling-Safe)

Standard softmax requires seeing all scores before normalizing. FlashAttention uses the online softmax algorithm to update incrementally per tile:

For each new K-block tile:
  m_new = max(m_old, max(S_tile)) // update running max
  l_new = exp(m_old-m_new) × l_old + sum(exp(S_tile-m_new)) // rescale + accumulate
  O = (l_old/l_new) × exp(m_old-m_new) × O_old + (1/l_new) × exp(S_tile-m_new) × V_tile
This produces the exact same result as standard softmax — no approximation.
