Why DiskANN? — The Memory Wall Problem

At billion-scale, in-memory indexes like HNSW need terabytes of RAM. Compressed indexes (IVF) save memory but sacrifice recall. DiskANN achieves the best of both worlds: high recall with SSD + small RAM footprint.

Algorithm

Vamana Graph Algorithm

Vamana builds a navigable graph with a single global entry point. The RobustPrune procedure controls the trade-off: α=1 produces a sparse graph (like a greedy spanner), while α>1 adds long-range edges that reduce graph diameter and improve search speed.

Architecture

SSD Index Architecture

DiskANN uses a two-tier design: compressed PQ codes in DRAM for distance estimation, full vectors + graph co-located on SSD for precise search. BeamSearch issues W parallel SSD reads per hop, piggybacking full vectors for implicit re-ranking.

DRAM Tier

PQ Compressed Vectors
• ~32 bytes per vector
• 1B vectors = ~32 GB RAM
Global Medoid (entry point)
• Single start node for all queries

SSD Tier

Full Vectors + Graph Edges
• Co-located in 4KB sectors
• One SSD read = node + all neighbors
BeamSearch (W=4)
• 4 parallel async SSD reads per hop
