Architecture System Architecture Overview

Mooncake separates prefill and decode into independent, elastically-scalable pools coordinated by a global Conductor scheduler. A distributed KV cache pool spanning GPU VRAM, CPU DRAM, and NVMe SSD enables aggressive cache reuse across requests.

Mooncake production architecture deployed at Kimi (Moonshot AI)

Conductor

Global scheduler with prefix-hash routing, load balancing, and cache-aware request placement.

Prefill Pool

Compute-optimized nodes for prompt processing. Supports incremental prefill to skip cached tokens.

Decode Pool

Memory-bandwidth-optimized nodes for autoregressive token generation with received KV caches.

KV Cache Pool

Distributed 3-tier storage across all nodes. GPUDirect RDMA enables fast cross-node KV transfers.

Storage Three-Tier KV Cache Pool

Mooncake's distributed KV cache pool organizes storage into three tiers with automatic promotion and demotion of KV blocks based on access frequency.

Hot KV entries are promoted up; cold entries are demoted down

Why three tiers? GPU VRAM is fastest but scarce (~40GB per GPU). CPU DRAM offers 6x more capacity at 1/20th the bandwidth. NVMe SSD adds another 16x capacity for cold KV entries that may be reused later. Together, they let Mooncake cache far more KV data than VRAM-only systems.

Scheduling Cache-Aware Scheduling

The Conductor hashes the prompt prefix and evaluates cache hit ratios on all prefill nodes, then routes the request to the node with the highest overlap — dramatically reducing time-to-first-token.

Cache-aware routing saves compute: An 85% cache hit means only 15% of the prompt needs prefill computation, translating to roughly 6x faster time-to-first-token compared to a cold start.
