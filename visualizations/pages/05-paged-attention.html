Problem KV Cache Memory Waste

In traditional LLM serving systems, KV cache memory is pre-allocated contiguously for each request up to the maximum sequence length. Since actual generation lengths are unpredictable, this leads to massive memory waste — often 60-80% of allocated GPU memory is unused.

Three Types of Waste

Reservation Waste

Pre-allocate for max_seq_len (e.g., 2048) but the request only generates 128 tokens. 93% of reserved memory is never used.

Internal Fragmentation

Last allocated block is only partially filled with KV entries. The remainder of that block is wasted space.

External Fragmentation

Free memory exists across GPU but in small scattered chunks — not enough contiguous space for a new request.

Analogy OS Virtual Memory Mapping

PagedAttention borrows the core idea from operating system virtual memory: use a page table (block table) to map logical addresses to non-contiguous physical locations. The KV cache no longer needs contiguous memory.

Contiguous Allocation

  • Pre-allocate for max_seq_len per request
  • Reservation waste: 60%+
  • External fragmentation blocks new requests
  • Cannot share memory between sequences

Advanced Beam Search with Copy-on-Write

In beam search, multiple beams share the same prefix. PagedAttention uses Copy-on-Write (CoW): beams share physical blocks via reference counting. A block is only copied when a beam needs to write different tokens to it.
