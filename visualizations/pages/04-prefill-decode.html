Phase Model Two-Phase Generation

LLM inference is split into two fundamentally different phases, each with distinct computational characteristics and hardware bottlenecks.

Prefill Phase

  • Parallel processing — all prompt tokens processed simultaneously in a single forward pass
  • Compute-bound — dominated by large matrix multiplications across all tokens
  • Builds the KV cache — generates Key and Value tensors for every layer and stores them for reuse
  • Produces the first token — the output of prefill is the probability distribution for token #1

Decode Phase

  • One token at a time — each forward pass generates exactly one new token
  • Memory-bound — dominated by reading model weights and KV cache from memory
  • Appends to KV cache — each new token adds its K and V vectors to the existing cache
  • Runs until EOS — generation continues until the end-of-sequence token or max length

Deep Dive KV Cache

The KV (Key-Value) cache is the central mechanism that makes autoregressive decoding efficient. Without it, every decode step would need to recompute attention over all previous tokens.

Without KV Cache

  • Each new token requires a full forward pass over all previous tokens
  • Redundant computation grows quadratically: O(n²)
  • Attention recomputed from scratch every step
  • Impractical for long sequences

With KV Cache

  • Previous K and V vectors are stored and reused
  • Only the new token's Q, K, V need to be computed
  • Each decode step is O(n) instead of O(n²)
  • Trades memory for massive compute savings
Cache Size = 2 × n_layers × seq_len × d_model × dtype_bytes
2
K + V — Two matrices are cached per layer: the Key matrix and the Value matrix. The Query is not cached because it's only needed for the current token.
n_layers
Number of transformer layers — Each layer has its own independent KV cache. Deeper models (more layers) need proportionally more cache memory. E.g., LLaMA-2 7B = 32 layers, 70B = 80 layers.
seq_len
Total sequence length — prompt tokens + generated tokens so far. This is the only parameter that grows during decode — each new token adds one row to every layer's K and V. This is why KV cache grows linearly with generation length.
d_model
Hidden dimension — The size of each K/V vector per token. Determined by the model architecture. E.g., 4096 for LLaMA-2 7B/13B, 8192 for 70B. With GQA, effective dim = n_kv_heads × d_head (often smaller than d_model).
dtype_bytes
Bytes per element — Depends on numerical precision. FP32 = 4 bytes, FP16/BF16 = 2 bytes (most common), INT8 = 1 byte, INT4 = 0.5 bytes. Quantizing KV cache (e.g., TurboQuant) can dramatically reduce memory.
Example (LLaMA-2 70B scale):
2 (K+V) × 32 layers × 4096 seq_len × 4096 d_model × 2 bytes (FP16)
= 2,147,483,648 bytes ≈ 2 GB per request

Analysis Compute vs Memory Bandwidth

Prefill and decode have opposite bottlenecks. Prefill saturates GPU compute with large matrix operations. Decode is starved for data, spending most time reading weights and cache from HBM.

Concepts TTFT & TBT — LLM Latency Metrics

Two key metrics define the user experience of LLM inference. Understanding how they are computed helps you estimate performance and identify bottlenecks.

TTFT — Time To First Token

The time from sending the request to receiving the first generated token. Dominated by the prefill phase.

TTFT ≈ Tprefill = nprompt × Cper_token / GPUFLOPS
Key factors:
• n_prompt — Number of input tokens (longer prompt = higher TTFT)
• C_per_token — FLOPs per token ≈ 2 × model_params (forward pass)
• GPU_FLOPS — GPU throughput (e.g., H100 = ~1000 TFLOPS FP16)
• Prefill is compute-bound — GPU utilization is high
Example: LLaMA-2 7B on A100
Prompt = 512 tokens
C_per_token = 2 × 7B = 14 GFLOPS
Total FLOPs = 512 × 14G = 7.168 TFLOPS
A100 FP16 = ~312 TFLOPS
TTFT ≈ 7168G / 312T ≈ 23 ms
(+ overhead: scheduling, KV write ≈ 40-50 ms real)

TBT — Time Between Tokens

The time between consecutive generated tokens during the decode phase. Determines the streaming speed perceived by users.

TBT ≈ Tdecode = Wmodel / BWmemory
Key factors:
• W_model — Model weight size in bytes (must read all weights per token)
• BW_memory — GPU HBM bandwidth (e.g., A100 = 2 TB/s)
• + KV cache read overhead grows with sequence length
• Decode is memory-bound — GPU compute is mostly idle
Example: LLaMA-2 7B on A100
FP16 weights = 7B × 2 bytes = 14 GB
A100 HBM BW = 2.0 TB/s
TBT ≈ 14 GB / 2.0 TB/s ≈ 7 ms
(+ KV cache read + softmax ≈ 10-15 ms real)
Total Generation Latency
Ttotal = TTFT + (noutput − 1) × TBT
Example: Generate 200 tokens with TTFT=45ms, TBT=12ms
Ttotal = 45 + (200 − 1) × 12 = 45 + 2388 = 2433 ms ≈ 2.4s   (83 tokens/sec)
