The Express Highway Analogy

HNSW organizes vectors into multiple layers, like a transit system. The top layer has few nodes with long-range "highway" connections. Each lower layer adds more nodes with shorter connections. Search starts at the top and descends greedily through layers to find nearest neighbors — just like taking a highway, then regional roads, then local streets.

Algorithm

Construction Algorithm

Each new node is assigned a random layer level using exponential decay. Most nodes live only on layer 0; a few reach the top. The node is then connected to M nearest neighbors at each layer from its assigned level down to 0, with excess neighbors pruned at Mmax.

Click "Insert Node" to see step-by-step insertion: layer assignment, greedy descent, neighbor connection, and pruning.
l = ⌊-ln(U(0,1)) × 1/ln(M)⌋  |  M = max connections per layer  |  Most nodes: layer 0, few nodes: top layers

Tuning

Key Parameters

Three parameters control HNSW's behavior: M (connections per node), ef (search beam width), and efConstruction (build-time beam width). Higher values improve recall but cost more memory and latency.

16 100 200

M (4-64)

Max bi-directional links per node per layer. Higher M = better recall, more memory. Typical: 12-48.

ef (10-500)

Search-time beam width (dynamic candidate list size). Higher ef = better recall, slower search. Typical: 64-200.

efConstruction (50-500)

Build-time beam width. Higher = better graph quality, slower build. Typical: 100-400. Set once, affects index quality.
Comparison

HNSW vs Vamana (DiskANN)

HNSW and Vamana represent two philosophies: hierarchical multi-layer vs flat single-layer with alpha-pruning. HNSW excels in-memory with online inserts; Vamana excels on SSD at billion scale with batch builds.
