Compare Four KV Cache Strategies

StreamingLLM was proposed at ICLR 2024. The key discovery: LLMs allocate anomalously high attention to initial tokens (attention sinks). These aren't semantically important — they are a softmax artifact. By keeping sink tokens plus a recent window, we get O(L) constant memory for infinite streaming.

🟣

Dense Attention

Full N x N attention over all tokens. Gold-standard quality but memory grows without bound.

Memory: O(N2) Compute: O(N2) Quality: Perfect
✘ OOM on long sequences
🟠

Window Attention

Keep only the last W tokens in the KV cache. Fixed memory, but quality collapses when context exceeds window.

Memory: O(W) Compute: O(W) Quality: Collapses
✘ Perplexity spikes at boundary
🔴

Sliding + Recompute

Re-encode the last W tokens from scratch each step. Correct quality but compute cost is prohibitive.

Memory: O(W) Compute: O(T × W) Quality: Correct
✘ Too slow for streaming
🟢

StreamingLLM

Keep 4 sink tokens + last W tokens. Constant memory, constant per-step compute, stable quality forever.

Memory: O(W + 4) Compute: O(W + 4) Quality: Stable
✔ Infinite streaming with constant memory

Discovery Attention Sink Phenomenon

Across all layers, initial tokens receive disproportionately high attention — even though they carry no special semantic meaning. This is a softmax artifact: softmax must distribute probability mass somewhere, and initial tokens become default "sinks" for unused attention.

Attention heatmaps for layers 1, 6, 12, and 24. Column 0 (first token) is consistently bright across ALL layers.

Animation Streaming KV Cache Window

The KV cache holds exactly 4 sink tokens (always the first 4) plus a sliding window of W recent tokens. As new tokens arrive, the oldest token in the recent window is evicted — but sinks are never touched.

Sinks [0-3] + Recent [4 ... 11]

Performance Perplexity Across Sequence Length

Dense attention works until it runs out of memory. Window attention holds steady within its window, then perplexity spikes catastrophically when context overflows. StreamingLLM remains flat and stable throughout.

Why re-map? Without re-mapping, position gaps (0,1,2,3 then 15,16,...) confuse RoPE — the model expects contiguous position IDs. Re-mapping to [0..9] restores positional coherence and maintains generation quality.

Summary Key Insight

KV Cache = Sink Tokens [0..3] + Recent Window [t-W+1 .. t]

Constant memory O(W + S) where S = 4 sink tokens, W = window size. Stable perplexity on sequences of 4M+ tokens.
