## 一、解决什么问题？

LLM推理中KV cache的内存占用随序列长度线性增长：

```
30B参数模型, batch=128, seq_len=1024:
KV Cache = 180 GB！

→ 一块A100 80GB根本装不下
→ 即使装得下，内存越大 → batch越小 → GPU利用率越低 → 吞吐量低
```

**核心问题：能否只保留一小部分KV cache，丢弃其余的，而不影响生成质量？**

## 二、三个关键发现

### 发现1：Attention矩阵极度稀疏

```
即使模型是"密集训练"的，推理时attention矩阵超过95%的值接近0！

  Layer 0:  95%+ 稀疏
  Layer 10: 96%+ 稀疏
  Layer 20: 97%+ 稀疏
  ...
  几乎所有层都是如此

→ 理论上，只需5%的KV cache就能解码出相同的token
→ 潜在20×内存节省！
```

### 发现2：Heavy Hitters (H₂) — 少数token贡献大部分注意力

```
将所有token的累计注意力分数排序，发现遵循幂律分布：

  累计注意力分数
  ▲
  █
  █
  █
  █ █
  █ █
  █ █ █
  █ █ █ █
  █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ █ ........
  └──────────────────────────────────────→ Token排名
   少数H₂   大量低分token

→ 少数token（Heavy Hitters）贡献了大部分注意力权重
→ 这些H₂与文本中高频共现的词强相关
→ 删除H₂ → 模型功能严重损坏
→ 删除非H₂ → 几乎不影响！
```

### 发现3：贪心的局部统计就够了

```
理想方案: 知道未来所有token的注意力分布 → 选最优的KV子集（Belady算法）
问题: 我们不知道未来！

惊喜发现: 只用"已看到的token的累计注意力分数"来决定保留哪些KV，
          效果与知道未来的oracle几乎一样好！

→ 不需要预测未来，简单的贪心策略就接近最优
```

## 三、H₂O算法

### 核心策略：保留 Heavy Hitters + 最近的token

```
KV Cache (固定大小 k):
┌──────────────────┬──────────────────┐
│   Heavy Hitters  │   Recent Tokens  │
│  (累计注意力最高)  │   (最近的token)   │
└──────────────────┴──────────────────┘
       50%                 50%

→ H₂ 提供"全局重要信息"
→ Recent 提供"局部上下文"
→ 两者缺一不可
```

### 淘汰算法

```
Algorithm: H₂ Eviction

每生成一个新token:
  1. 将新token加入cache
  2. if cache满了:
       a. 计算cache中所有token的累计注意力分数
          F_score(token) = Σ (该token在所有历史step中获得的注意力)
       b. 候选淘汰集 = cache - {最近L个token} - {新token}
       c. 淘汰累计注意力分数最低的token
  3. 继续解码

关键细节:
  - 淘汰时不交换内存，直接在原位填入新KV → IO高效
  - 累计分数是增量更新的，不需要重新扫描历史
```

### 图解示例

```
Cache大小 = 3, 输入: "Children laughed and played in the sunny park"

Step 4 (解码"in"):
  Cache: [Children, laughed, played]   分数: [1.4, 1.5, 1.8]
  新token "in" 要加入
  → 淘汰分数最低的 "Children" (1.4)
  Cache: [laughed, played, in]

Step 5 (解码"the"):
  Cache: [laughed, played, in]   分数: [1.5, 2.7, 0.6]
  新token "the" 要加入
  → 淘汰分数最低的 "in" (0.6)
  Cache: [laughed, played, the]

→ "played" 作为heavy hitter一直被保留
→ 低分token被自然淘汰
```

## 四、理论保证

H₂O不只是启发式方法，论文给出了理论证明：

```
假设attention是次模函数(submodular)，则：

  H₂O的贪心策略生成的集合 S̃ᵢ 满足:
  f(S̃ᵢ) ≥ (1-α)(1-1/e) · max_{|S|=k} f(S) - β

其中 α,β > 0 是小常数

→ 这是经典次模最大化的 (1-1/e) ≈ 63% 近似比
→ 贪心策略理论上接近最优！
```

## 五、性能结果

### 内存节省 vs 质量保持

```
只用20%的KV cache预算，H₂O在大多数任务上匹配完整KV cache的表现：

OPT-30B, 20% KV budget:
  Full KV:  COPA 85.00, OpenBookQA 43.20, PiQA 78.51, Winogrande 70.24
  H₂O:     COPA 79.22, OpenBookQA 43.80, PiQA 85.00, Winogrande 71.67
                                          ↑ 有些任务H₂O反而更好！(正则化效应)

对比其他策略 (20% budget, OPT-30B):
  Local (只保留最近):        COPA 48.00, OpenBookQA 25.20
  Sparse Transformer:       COPA 50.00, OpenBookQA 24.60
  Local + H₂:              COPA 84.00, OpenBookQA 43.00  ← H₂的加成效果巨大
```

### 吞吐量提升

| 系统 | OPT-6.7B (T4 GPU) | OPT-30B (T4 GPU) |
|------|-------------------|-------------------|
| HF Accelerate | 20.4 tok/s | 0.6 tok/s |
| DeepSpeed | 10.2 tok/s | 0.6 tok/s |
| FlexGen | 20.2 tok/s | 8.1 tok/s |
| **H₂O (20%)** | **35.1 tok/s** | **12.7 tok/s** |

→ H₂O比FlexGen快**1.7×**，比DeepSpeed快**3.4×**，比Accelerate快**1.7×**

原因：KV cache变小 → batch size变大 → GPU利用率提高

### A100 GPU上的效果

| 配置 (OPT-6.7B) | FlexGen | H₂O (20%) |
|----------------|---------|-----------|
| 2048+2048, batch=24 | 494.1 tok/s | **918.9 tok/s** |
| 2048+2048, batch=64 | OOM | **1161.0 tok/s** |

→ FlexGen在batch=64时OOM，H₂O可以继续跑因为KV cache只有20%

### 与StreamingLLM的结合

```
H₂O + StreamingLLM的sink token思路 → 更好的流式性能

在4百万token的流式输入上:
  StreamingLLM原版:  perplexity偏高
  StreamingLLM + H₂O: perplexity更低 ← heavy hitters提供更好的信息保留
```

### 与量化的兼容性

```
H₂O + 量化(INT4/INT8) = 双重压缩！

  单独H₂O:  5× 内存节省
  单独量化:  2-4× 内存节省
  H₂O+量化: 10-20× 内存节省

而且组合的准确率通常比单独使用任一方法更高（惊喜发现）
```

## 六、与StreamingLLM的对比

```
StreamingLLM:
  保留: 固定的前4个sink tokens + 最近L个tokens
  淘汰: 中间的全部丢弃
  优点: 极其简单，零开销
  缺点: 丢失中间所有信息

H₂O:
  保留: 动态选择的Heavy Hitters + 最近的tokens
  淘汰: 累计注意力最低的token
  优点: 保留全局重要信息，质量更好
  缺点: 需要维护注意力分数统计

关系: 两者互补
  StreamingLLM的sink token ≈ H₂O发现的H₂的特例
  H₂O可以看作StreamingLLM的"智能升级版"
```

## 八、一句话总结

> **H₂O发现LLM的注意力矩阵虽然密集训练但推理时超过95%稀疏，少数"Heavy Hitter"token贡献了大部分注意力权重。据此提出只保留20%的KV cache（Heavy Hitters + 最近token），用简单贪心策略实现接近理论最优的淘汰决策，在几乎不损失质量的前提下将吞吐量提升最高29倍，奠定了KV cache智能淘汰的研究方向。**
