## 一、摘要（中文翻译）

大语言模型（LLM）的高吞吐量服务需要同时批处理足够多的请求。然而，现有系统在这方面面临困难，因为每个请求的键值缓存（KV cache）占用大量内存，且会动态增长和缩小。当内存管理效率低下时，碎片化和冗余复制会导致大量内存浪费，从而限制了批处理大小。

为解决这一问题，我们提出了 **PagedAttention**——一种受操作系统经典虚拟内存和分页技术启发的注意力算法。在此基础上，我们构建了 **vLLM**，一个LLM服务系统，实现了：（1）KV cache内存的近零浪费；（2）请求内部及请求之间灵活的KV cache共享，进一步降低内存使用。

评估显示，vLLM在保持相同延迟水平的情况下，将主流LLM的吞吐量提升了 **2-4倍**，优于FasterTransformer和Orca等先进系统。在更长序列、更大模型和更复杂解码算法下，改进效果更加显著。

## 二、问题背景

### 2.1 为什么KV Cache是瓶颈？

以13B参数的OPT模型在NVIDIA A100 40GB GPU上运行为例：

| 内存用途 | 占比 |
|---------|------|
| 模型参数（静态） | ~65%（26GB） |
| KV Cache（动态） | >30% |
| 激活值等（临时） | <5% |

- **单个token的KV cache大小**（OPT-13B）= 2 × 5120 × 40 × 2 bytes = **800KB**
- **单个请求最大KV cache**（2048 tokens）= **1.6GB**
- GPU内存仅能容纳几十个并发请求，内存管理不善会进一步缩小批处理规模

### 2.2 现有系统的三大内存浪费

现有系统（如FasterTransformer、Orca）将每个请求的KV cache存储在**连续内存空间**中，导致三种浪费：

1. **预留浪费（Reservation）**：按最大可能序列长度预分配内存，实际生成长度往往远小于上限
2. **内部碎片（Internal Fragmentation）**：请求结束后才知道实际长度，预分配的多余空间全部浪费
3. **外部碎片（External Fragmentation）**：不同请求预分配大小不同，产生无法利用的内存间隙

**实测数据**：现有系统中，实际有效利用的KV cache内存仅占 **20.4% - 38.2%**，超过60%的内存被浪费。

## 三、核心创新：PagedAttention

### 3.1 核心思想：像操作系统一样管理内存

PagedAttention 的灵感直接来源于操作系统的**虚拟内存分页机制**：

| 操作系统概念 | PagedAttention 对应 |
|------------|-------------------|
| 页（Page） | KV Block（KV块） |
| 字节（Byte） | Token |
| 进程（Process） | 请求（Request） |
| 页表（Page Table） | Block Table（块表） |
| 虚拟地址空间 | 逻辑KV块序列 |
| 物理内存页帧 | 物理KV块 |

### 3.2 算法原理

传统注意力机制要求KV cache存储在连续内存中。PagedAttention 将KV cache分割成固定大小的**KV块**（每块包含固定数量token的key和value向量），这些块**无需存储在连续物理内存中**。

注意力计算被转化为**分块计算**：

- 将所有KV cache分成多个block：K_j, V_j
- 对每个block独立计算注意力分数 A_ij
- 最终通过加权汇总得到输出

关键点：**PagedAttention内核在计算时，会独立地从不同物理位置获取和处理各个KV块**，无需它们在物理上连续。

### 3.3 KV Cache Manager（KV缓存管理器）

vLLM 的内存管理器核心设计：

1. **按需分配**：不预留最大长度空间，仅在需要时分配新的物理块
2. **块表映射**：维护逻辑块到物理块的映射表（类比页表）
3. **从左到右填充**：新token填入最后一个逻辑块，满了才分配新块
4. **内存浪费降至最低**：每个请求最多浪费一个块的空间（最后一个未填满的块）

## 四、关键技术创新点

### 4.1 创新一：Copy-on-Write（写时复制）机制

**应用场景**：并行采样（Parallel Sampling）——同一个prompt生成多个不同输出

- 多个输出序列**共享prompt部分的物理KV块**
- 每个物理块维护**引用计数**
- 当某个序列需要修改共享块时，才复制该块（Copy-on-Write）
- 大幅节省长prompt场景下的内存

**实际效果**：在程序助手等场景中，prompt的KV cache约占总KV cache的12%，通过共享可显著减少内存。

### 4.2 创新二：Beam Search 的高效内存共享

传统系统在beam search中需要频繁**复制大量KV cache**。vLLM的方案：

- 不同beam候选可以在block粒度上共享物理块
- 共享模式随解码推进**动态变化**（类似OS中fork创建的进程树）
- 仅当新生成的token需要写入旧的共享块时，才触发copy-on-write
- 最多只复制一个block的数据

**实际效果**：beam search可节省高达 **55%** 的内存。

### 4.3 创新三：共享前缀（Shared Prefix）

- LLM服务中，许多请求共享相同的系统提示（system prompt）
- vLLM预先缓存共享前缀的KV cache物理块
- 新请求直接将逻辑块映射到缓存的物理块
- prompt计算只需处理用户特定的任务输入部分

**类比**：类似操作系统中多个进程共享动态链接库。

### 4.4 创新四：抢占与调度机制

当GPU内存不足时，vLLM提供两种策略：

| 策略 | 机制 | 适用场景 |
|-----|------|---------|
| **Swapping（交换）** | 将KV块从GPU复制到CPU内存 | CPU-GPU带宽充足时 |
| **Recomputation（重计算）** | 释放KV cache，需要时重新计算 | GPU计算能力强时 |

- 采用 **FCFS（先来先服务）** 调度策略
- 实施 **全有或全无** 驱逐策略（序列的所有块一起驱逐）
- 序列组（如同一beam search的候选）**联合调度**

### 4.5 创新五：分布式执行

- 支持 Megatron-LM 风格的张量模型并行
- 采用 SPMD（单程序多数据）执行模式
- **集中式调度器** 协调所有GPU worker
- KV cache管理器通过指令控制各worker的物理块分配
- 各GPU worker上的KV cache管理**完全同步**，无需额外通信

## 五、实验结果

### 5.1 模型配置

| 模型 | GPU配置 | 参数大小 | KV Cache可用内存 |
|-----|---------|---------|----------------|
| OPT-13B | 1×A100 40GB | 26GB | 12GB |
| OPT-66B | 4×A100 40GB | 132GB | 21GB |
| OPT-175B | 8×A100 80GB | 346GB | 264GB |

### 5.2 吞吐量对比

与当时最先进的系统对比：

| 对比维度 | vLLM vs FasterTransformer | vLLM vs Orca |
|---------|--------------------------|--------------|
| OPT-13B | **最高14-24倍** | **最高2.2倍** |
| OPT-175B | **最高2.2倍** | **最高2.7倍** |

### 5.3 内存利用率

| 系统 | 有效内存利用率 |
|-----|-------------|
| Orca (Max) | 20.4% |
| Orca (Pow2) | 26.8% |
| Orca (Oracle) | 38.2% |
| **vLLM** | **96.3%** |

vLLM将KV cache的有效内存利用率从不到40%提升到了 **96.3%**，几乎是零浪费。

## 六、核心思想总结

### 跨学科的系统创新

本文最大的创新在于**将操作系统领域数十年的成熟技术（虚拟内存、分页、写时复制、交换）迁移到了LLM推理的KV cache管理中**。这种跨学科的思维方式带来了巨大的性能提升：

1. **分页（Paging）** → 消除内存碎片，实现灵活的非连续内存管理
2. **虚拟内存映射** → 逻辑块到物理块的映射，按需分配
3. **写时复制（Copy-on-Write）** → 高效的KV cache共享（并行采样、beam search）
4. **交换（Swapping）** → GPU内存不足时将KV块卸载到CPU
5. **共享库** → 前缀共享，减少冗余计算和存储

### 为什么这篇论文如此重要？

1. **奠基性**：vLLM已成为LLM推理服务的事实标准，几乎所有后续的LLM serving系统都建立在PagedAttention的基础之上
2. **开源影响力**：vLLM项目在GitHub上获得了巨大关注，被广泛应用于工业界和学术界
3. **思维范式**：开创了"用系统思维解决AI问题"的范式，启发了大量后续工作（如Mooncake的分离式架构、SGLang的RadixAttention等）
4. **实际效果**：2-4倍的吞吐量提升在实际部署中意味着**成本降低50-75%**

### 一句话概括

> **PagedAttention 将操作系统的虚拟内存管理思想引入LLM推理，通过分页、写时复制和交换机制，将KV cache内存利用率从20%提升至96%，实现了2-4倍的吞吐量提升，彻底改变了LLM推理服务的技术范式。**
